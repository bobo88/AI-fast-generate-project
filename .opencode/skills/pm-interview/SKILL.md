## Purpose

用于系统性评估机器人硬件 + 软件公司产品经理候选人的真实水平。

目标不是判断候选人“会不会说”，而是判断：

- 是否真正做过产品
- 是否理解机器人 / IoT / 软件系统的本质
- 是否具备结构化思考能力
- 是否具备需求拆解、优先级、推进、跨部门协作能力
- 是否具备从 0 到 1、从 1 到 100 的产品能力
- 是否适合当前公司的阶段和业务

## Interview Principle

禁止只问“你懂不懂”“做过没有”。

必须通过：

```md
经历 → 细节 → 决策 → 数据 → 复盘 → 追问
```

来验证候选人是否真实做过。

核心原则：

1. 所有问题必须能继续向下追 3 层
2. 每个能力点，都要有真实案例验证
3. 不看候选人“讲得多好”，看是否：

   - 讲得清
   - 有逻辑
   - 有数字
   - 能说出为什么

4. 不要只判断“优秀不优秀”，而是判断：

   - 强项是什么
   - 短板是什么
   - 是否适合当前团队

## Candidate Evaluation Dimensions

必须从下面 10 个维度评估。

## 1. 产品基础能力

重点判断：

- 是否真的理解“产品经理”的职责
- 是否能把一个模糊需求变成明确方案
- 是否会写 PRD、流程图、原型图
- 是否知道需求、功能、方案、目标之间的区别

### 必问问题

```md
1. 你认为产品经理最重要的能力是什么？为什么？
2. 你做过的最复杂的产品是什么？请从背景、目标、用户、功能、结果完整讲一遍。
3. 当老板只告诉你“做一个远程控制机器人功能”，你会怎么开始？
4. 你通常如何整理需求？
5. 一个需求从提出到上线，你具体会怎么推进？
```

### 继续追问

```md
- 为什么这么设计？
- 有哪些备选方案？
- 最终怎么选择？
- 哪个地方最难？
- 最后结果如何验证？
```

### 判断标准

#### 初级

- 只会描述功能
- 不会讲目标、用户、场景

#### 中级

- 能讲清楚完整流程
- 会拆需求
- 能说出优先级

#### 高级

- 能从业务、用户、技术、成本一起考虑
- 能讲清楚为什么这样设计
- 能提前想到风险和未来扩展

## 2. 机器人 / 硬件理解能力

你们公司是机器人硬件 + 软件公司，因此必须判断候选人是否真的理解硬件产品。

很多互联网产品经理的问题是：

- 只懂 App
- 不懂硬件限制
- 不懂传感器、网络、延迟、电池、成本、量产

### 必问问题

```md
1. 你是否做过和硬件、IoT、机器人相关的产品？
2. 如果让你设计一个可以远程控制的小车机器人，你会考虑哪些模块？
3. 摄像头、云台、电机、电池、网络、GPS、避障分别会影响哪些产品能力？
4. 如果用户反馈“机器人有时候控制会延迟 2 秒”，你会怎么排查？
5. 机器人在家里断网了，产品上应该怎么设计？
6. 如果机器人电量只剩 10%，系统应该如何处理？
```

### 候选人应该至少提到

- 网络延迟
- 弱网场景
- 离线机制
- 电量不足
- 自动回充
- 设备状态同步
- 摄像头与视频流
- 硬件故障
- 固件升级
- OTA
- 安全性

### 高级追问

```md
假设现在要做一个家庭巡逻机器人：

- 可以远程查看监控
- 可以移动
- 可以自动巡逻
- 可以异常报警

请你拆出：

1. 用户场景
2. 核心功能
3. MVP
4. 后续版本路线
5. 最大风险
```

看候选人是否能够把硬件、软件、用户场景整合起来。

## 3. 需求拆解与优先级

### 必问问题

```md
1. 现在有 20 个需求，但开发资源只够做 5 个，你怎么决定？
2. 如果老板、客户、研发都在提不同需求，你怎么平衡？
3. 一个功能用户想要，但技术成本很高，你怎么处理？
4. 你通常如何定义 MVP？
```

### 希望候选人提到

- 用户价值
- 商业价值
- 技术成本
- 实现难度
- 时间成本
- 风险
- ROI
- Kano
- 四象限
- RICE

### 验证是否真实

继续问：

```md
请举一个你真的砍掉过的需求。
为什么砍？
最后谁不满意？
你怎么说服他们？
```

## 4. 逻辑与结构化思维

### 测试方法

现场给一个模糊题。

```md
我们准备做一个机器人远程看家产品，请你现场拆解。
```

观察候选人是否能主动拆成：

- 用户是谁
- 场景是什么
- 核心问题是什么
- 主要功能是什么
- MVP 是什么
- 风险是什么

### 高级标准

优秀候选人会主动输出：

```md
用户
→ 场景
→ 问题
→ 需求
→ 功能
→ 流程
→ 风险
→ 数据指标
```

如果候选人一上来就直接说功能，多半不够成熟。

## 5. 数据与结果导向

重点判断：

- 是否真的关心结果
- 是否知道如何衡量成功

### 必问问题

```md
1. 你做过的产品里，最重要的指标是什么？
2. 你如何判断一个功能是否成功？
3. 如果一个新功能上线后，用户并没有使用，你会怎么办？
4. 你会看哪些数据？
```

机器人相关可重点问：

- 设备在线率
- 日活
- 使用时长
- 控制成功率
- 视频观看时长
- 报警触发率
- 留存率
- 复购率
- 故障率

### 继续追问

```md
你之前项目中，有没有哪个指标没有达到目标？
为什么？
你做了什么？
最后改善了吗？
```

## 6. 跨团队协作与推进

机器人公司通常涉及：

- 产品
- 硬件
- 嵌入式
- 后端
- 前端
- App
- 测试
- 供应链
- 售后

因此必须判断候选人是否能推进复杂项目。

### 必问问题

```md
1. 你合作过最难沟通的研发是什么情况？
2. 研发说做不了，你怎么办？
3. 硬件延期、软件延期、测试延期时，你如何处理？
4. 如果两个部门意见完全冲突，你怎么推动？
```

### 重点看

- 是否会推动
- 是否会对齐目标
- 是否会拿数据和事实说话
- 是否只会“催”

## 7. 项目管理与执行

### 必问问题

```md
1. 你通常如何排期？
2. 如何保证项目按时上线？
3. 如果已经快上线了，但发现有一个严重问题，你会怎么办？
4. 你是否做过跨月、跨季度的大项目？
```

### 追问

```md
- 具体有哪些里程碑？
- 谁负责什么？
- 哪个节点最容易延期？
- 如何提前发现风险？
```

## 8. 真实水平验证法

为了避免候选人“背答案”，必须使用以下方法。

### 方法一：深挖经历

```md
你说你做过 XX 功能。
那请你：

- 当时用户是谁？
- 最开始的问题是什么？
- 你亲自做了什么？
- 研发多少人？
- 做了多久？
- 最后结果是什么？
```

如果候选人答不出来，大概率不是核心参与者。

### 方法二：让候选人现场设计

```md
请你在 10 分钟内，现场设计一个“机器人远程看护系统”。
```

要求输出：

- 用户
- 场景
- 功能
- 页面
- 流程
- 风险
- 指标

看其结构化能力。

### 方法三：反向验证

```md
你刚才说这个功能很成功。
那它失败的可能性是什么？
如果重做一次，你会改什么？
```

真正做过的人，一定知道缺点和问题。

## 9. 面试评分表

建议每项 1~5 分。

```md
1. 产品基础能力
2. 结构化思维
3. 硬件 / 机器人理解
4. 需求拆解
5. 优先级能力
6. 数据意识
7. 沟通推进
8. 项目管理
9. 学习能力
10. 与公司匹配度
```

### 最终判断

```md
4.5-5 分：非常强，可以做核心 PM
3.5-4.5 分：能独立负责模块
2.5-3.5 分：需要培养
2.5 以下：不建议录用
```

并且最终必须输出：

```md
候选人优点：
候选人短板：
最适合负责什么：
最大风险：
是否建议录用：
建议岗位级别：
```

## Advanced Version

如果候选人是高级产品经理 / 产品总监，还必须增加：

- 商业模式能力
- 产品路线图能力
- 团队管理能力
- 从 0 到 1 与从 1 到 100 的区别
- 如何决定做什么、不做什么
- 如何带领多个产品经理
- 如何管理复杂项目和资源冲突

高级候选人必须能回答：

```md
如果公司未来要做：

- 家庭机器人
- 巡检机器人
- 机器人 SaaS 平台
- 设备管理后台
- 多端 App

你会如何规划未来 1 年的产品路线？
```

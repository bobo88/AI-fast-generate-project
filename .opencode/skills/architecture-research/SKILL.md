# architecture-research

## Purpose

用于系统性地研究、拆解、设计、评估、沉淀“架构”相关内容。

适用于：

- 企业系统架构
- 微服务架构
- 单体架构
- SaaS / 多租户架构
- SSO / 权限架构
- 高并发架构
- AI Agent / MCP / Skill 架构
- IoT / 机器人 / ROS2 架构
- 前后端分离架构
- 游戏服务平台架构
- 数据平台 / 监控平台架构
- 组织与团队架构

目标不是简单输出“怎么做”，而是：

1. 看清架构的本质
2. 搞清楚为什么这样设计
3. 明确有哪些方案、各自适用场景、优缺点
4. 输出可复用、可沉淀、可扩展的架构方法论
5. 最终形成“架构知识库 + 架构模板 + 架构决策模型”

注意：需要把架构设计的 md 文档，放到一个统一的目录`architecture`下，方便后续的维护和查阅。

## Core Thinking

任何架构问题，都必须按照下面 8 个层次进行分析。

### 1. 背景与目标

必须先明确：

- 为什么要做？
- 当前存在什么问题？
- 谁是核心用户？
- 业务目标是什么？
- 技术目标是什么？
- 成本、时间、团队规模限制是什么？

输出格式：

```md
### 背景

...

### 当前问题

...

### 架构目标

- 功能目标
- 性能目标
- 安全目标
- 可维护目标
- 可扩展目标
```

### 2. 系统边界

必须拆清楚系统由哪些部分组成。

要求输出：

- 系统上下游
- 外部依赖
- 核心模块
- 不属于当前系统的部分
- 数据流与职责边界

输出格式：

```md
### 系统边界

输入：
输出：
外部系统：
内部模块：
不包含范围：
```

### 3. 架构分层

所有系统都必须拆成至少 5 层：

1. 用户层
2. 接入层
3. 业务层
4. 数据层
5. 基础设施层

输出格式：

```md
### 架构分层

#### 用户层

- Web
- H5
- App
- 管理后台

#### 接入层

- Nginx
- Gateway
- CDN
- WAF

#### 业务层

- 用户中心
- 订单中心
- 权限中心
- 支付中心

#### 数据层

- MySQL
- Redis
- ES
- MQ

#### 基础设施层

- Docker
- Kubernetes
- Prometheus
- CI/CD
```

并要求进一步输出：

- 每层职责
- 每层与上下层如何交互
- 哪些能力应该下沉
- 哪些能力不能跨层调用

### 4. 核心模块设计

每个核心模块必须输出：

```md
#### 模块名称

职责：
为什么存在：
输入：
输出：
依赖：
状态：
核心数据：
风险：
扩展点：
```

例如：

- 用户中心
- SSO 单点登录
- 权限系统
- 支付系统
- 商品系统
- Agent / Skill 系统
- 监控告警系统
- 消息通知系统

### 5. 数据与流程

要求输出两部分：

#### 5.1 数据结构

必须输出：

- 核心实体
- ER 关系
- 主键、外键
- 状态流转
- 是否需要分库分表

输出格式：

```md
### 数据模型

用户 User
订单 Order
商品 Product
权限 Permission
角色 Role
```

#### 5.2 业务流程

必须输出：

- 正常流程
- 异常流程
- 边界场景
- 时序图 / 流程图

输出格式：

```md
1. 用户发起请求
2. 网关校验 Token
3. 进入业务服务
4. 写数据库
5. 发消息通知
```

### 6. 架构决策

架构最重要的不是“会不会”，而是“为什么这么选”。

每一个关键点，都必须输出 ADR（Architecture Decision Record）：

```md
### 架构决策

问题：
有哪些可选方案：

- 方案 A
- 方案 B
- 方案 C

最终选择：
原因：
优点：
缺点：
未来什么时候需要升级：
```

必须特别分析：

- 为什么选择单体 / 微服务
- 为什么选择 MySQL / PostgreSQL
- 为什么选择 Redis
- 为什么选择 JWT / Session
- 为什么选择 Go / Node / Java
- 为什么选择 Vue / React / Nuxt
- 为什么选择 Kafka / RabbitMQ
- 为什么选择 Docker / K8s

### 7. 风险与演进

任何架构都不是一次性设计完成的。

必须输出：

```md
### 当前风险

- 性能风险
- 安全风险
- 人员风险
- 业务风险

### 演进路线

阶段 1：0-1000 用户
阶段 2：1000-10 万 用户
阶段 3：10 万-100 万 用户
阶段 4：百万级以上
```

重点分析：

- 什么地方会先崩
- 哪个模块最容易成为瓶颈
- 哪些能力应该预留扩展点
- 什么时候需要拆服务
- 什么时候需要引入缓存、消息队列、分布式

### 8. 输出沉淀

最终必须把所有架构研究沉淀成下面 6 类成果：

```md
1. 架构图
2. 模块说明书
3. 数据模型
4. API 文档
5. 技术选型说明
6. 架构决策记录 ADR
```

最终还要进一步沉淀：

- 标准化项目模板
- 最佳实践
- 常见错误清单
- Review 检查项
- 后续可自动生成的代码模板

## Mandatory Output Structure

以后每次使用 `/architecture-research`，必须按下面结构输出：

```md
## 主题

### 一、背景与目标

### 二、系统边界

### 三、架构分层

### 四、核心模块设计

### 五、数据与流程

### 七、风险与演进

### 八、最终架构方案

```

## Example Invocation

```md
/architecture-research 设计一个支持多个系统统一登录、支持代理商和消费者绑定、支持支付、支持后台管理的游戏平台用户中心架构
```

Skill 输出时，应自动：

- 判断是否需要 SSO
- 判断是否需要权限系统
- 判断是否需要代理商体系
- 判断是否需要支付体系
- 判断是否需要多租户
- 判断未来是否会拆微服务
- 自动给出项目目录、数据库、接口、技术选型、部署方式

## Advanced Research Angles

作为资深架构师 / 架构专家，每次研究都要从以下 10 个角度重新审视：

1. 业务视角
2. 用户视角
3. 技术视角
4. 成本视角
5. 风险视角
6. 长期演进视角
7. 团队协作视角
8. 运维与监控视角
9. 安全与权限视角
10. 自动化与模板化视角

并进一步追问：

- 如果团队扩大 10 倍，这个架构还成立吗？
- 如果用户扩大 100 倍，会先出什么问题？
- 如果未来换技术栈，哪些地方最容易重构？
- 哪些能力应该做成平台？
- 哪些能力应该做成通用组件或 Skill？

## Final Goal

最终目标：

把“架构设计”从依赖个人经验，变成：

```md
业务输入
→ 架构分析
→ 模块拆解
→ 技术选型
→ 数据设计
→ 风险评估
→ 输出标准化架构方案
→ 自动生成模板 / 文档 / 代码
```

最终形成你自己的：

- 架构知识库
- 架构设计方法论
- 架构模板库
- 架构决策库
- 架构 Review 体系
- 自动化生成体系

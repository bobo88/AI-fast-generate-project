# SaaS 多租户架构最佳实践

## 主题：企业级 SaaS 多租户架构设计与最佳实践

---

### 一、背景与目标

#### 背景

通用 SaaS 平台需要服务大量租户（1000+），每个租户需要：
- 数据完全隔离（独立数据库）
- 配置独立（主题、功能开关）
- 计费独立（按租户结算）
- 权限独立（租户内角色管理）

#### 当前问题

1. **数据隔离** - 租户间数据必须严格隔离，不能有任何泄露风险
2. **资源管理** - 1000+ 租户需要高效管理，不能逐个维护
3. **成本控制** - 需要按租户计费、控制资源使用
4. **运维复杂** - 大量独立数据库，部署、监控、备份工作量大

#### 架构目标

| 目标类型 | 具体目标 |
|----------|----------|
| **功能目标** | 多租户数据隔离、租户管理、计费、权限 |
| **性能目标** | 支持 1000+ 租户、租户内万级用户 |
| **安全目标** | 租户间数据零泄露、权限严格管控 |
| **可维护目标** | 统一运维、自动化部署、快速扩缩容 |
| **可扩展目标** | 支持从 1000 到 10000+ 租户平滑演进 |

### 二、系统边界

#### 系统边界

| 类别 | 内容 |
|------|------|
| **输入** | 租户注册请求、租户配置、用户请求（带租户标识） |
| **输出** | 租户工作空间、租户报表、计费账单 |
| **外部系统** | 支付网关（Stripe/支付宝）、短信/邮件服务、第三方登录 |
| **内部模块** | 租户管理服务、租户配置服务、计费服务、权限服务、各业务服务 |
| **不包含范围** | 租户自有业务系统（仅提供平台能力） |

### 三、架构分层

#### 整体架构分层

```
#### 用户层

- 租户管理员 Portal
- 租户用户 Web App
- 租户用户 App (iOS/Android)
- 平台运营后台

#### 接入层

- CDN (静态资源)
- 负载均衡 (Nginx/LB)
- API Gateway (统一入口、租户识别)
- WAF (安全防护)

#### 业务层 (多租户服务)

├── 平台服务 (Platform)
│   ├── 租户管理服务 (Tenant Service)
│   ├── 租户配置服务 (Tenant Config Service)
│   ├── 计费服务 (Billing Service)
│   └── 运营后台服务 (Admin Service)
│
├── 租户公共服务 (Shared Services)
│   ├── 认证服务 (Auth - Tenant Context)
│   ├── 权限服务 (Permission - Tenant Scope)
│   ├── 消息服务 (Notification)
│   └── 文件服务 (File Service)
│
└── 业务服务 (Business Services)
    ├── 用户/组织服务 (支持多租户)
    ├── 订单/商品服务 (支持多租户)
    └── 自定义服务 (租户扩展)

#### 数据层

├── 平台数据库 (Platform DB)
│   ├── 租户元数据
│   ├── 计费数据
│   └── 运营数据
│
├── 租户数据库 (Tenant DBs)
│   ├── 每个租户独立 MySQL 实例
│   ├── 租户内数据完全隔离
│   └── 按租户分库分表
│
└── 共享缓存 (Redis Cluster)
    ├── 租户会话缓存
    └── 租户配置缓存

#### 基础设施层

├── Kubernetes (容器编排)
├── Tenant Operator (租户自动化管理)
├── Database Proxy (数据库代理)
├── CI/CD (多环境部署)
└──监控告警 (租户级别)
```

#### 每层职责与交互

| 层级 | 职责 | 关键能力 | 禁止跨层 |
|------|------|----------|----------|
| 用户层 | 多租户 UI 隔离 | 租户域名/子域名 | - |
| 接入层 | 租户识别、路由 | Tenant ID 注入 | 禁止无租户标识请求 |
| 业务层 | 租户上下文传递 | ThreadLocal/Context | 禁止跨租户查询 |
| 数据层 | 租户数据库隔离 | 数据库代理 | 禁止跨库 JOIN |
| 基础设施层 | 租户资源管理 | 自动创建/删除 | - |

### 四、核心模块设计

#### 4.1 租户管理服务

```
职责：租户生命周期管理、租户注册、租户开通
为什么存在：1000+ 租户需要自动化管理
输入：租户注册请求、企业信息
输出：租户实例、初始化数据库
依赖：Kubernetes、Docker Registry
核心数据：租户ID、企业信息、租户状态、套餐
风险：租户创建失败、资源泄露
扩展点：支持租户自助注册、租户迁移
```

#### 4.2 租户配置服务

```
职责：租户个性化配置、主题、功能开关
为什么存在：租户需要独立配置、UI定制
输入：租户配置变更
输出：租户配置、生效配置
依赖：Redis、租户数据库
核心数据：租户配置项、主题配置、功能开关
风险：配置泄露到其他租户
扩展点：支持配置版本、回滚
```

#### 4.3 计费服务

```
职责：租户套餐管理、用量统计、账单生成
为什么存在：按租户计费、控制资源
输入：租户套餐选择、用量数据
输出：租户账单、续费提醒
依赖：支付网关、计量服务
核心数据：套餐配置、租户用量、月账单
风险：计费不准、套利
扩展点：支持后付费、按量付费
```

#### 4.4 认证与权限服务

```
职责：多租户身份认证、租户内权限管理
为什么存在：租户内独立用户体系、安全管控
输入：租户用户登录、权限请求
输出：租户上下文 Token、权限校验结果
依赖：用户服务、租户数据库
核心数据：租户用户、角色、权限、租户ID
风险：租户隔离失效、权限泄露
扩展点：支持 SSO、第三方登录
```

#### 4.5 数据库代理 (Tenant Proxy)

```
职责：租户数据库连接管理、SQL 路由
为什么存在：1000+ 数据库需要统一管理
输入：SQL 请求 + Tenant ID
输出：路由到对应租户数据库
依赖：MySQL、连接池
核心数据：租户数据库映射、连接池配置
风险：连接泄露、SQL 注入
扩展点：支持读写分离、连接池共享
```

#### 4.6 Kubernetes Operator

```
职责：自动化租户资源管理（CRD）
为什么存在：手动管理 1000+ 租户资源不现实
输入：租户 CR (Custom Resource)
输出：自动创建 Namespace、数据库、配置
依赖：Kubernetes、Helm
核心数据：租户资源定义、状态同步
扩展点：支持租户迁移、灾备
```

#### 核心表设计

```sql
-- 平台数据库
-- 租户表
CREATE TABLE tenants (
  id BIGINT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  domain VARCHAR(100) UNIQUE,
  status VARCHAR(20), -- active/suspended/terminated
  package_id BIGINT,
  db_instance VARCHAR(100), -- 租户数据库实例名
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- 套餐表
CREATE TABLE packages (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50),
  max_users INT,
  max_storage BIGINT,
  price DECIMAL(10,2),
  features JSONB
);

-- 账单表
CREATE TABLE bills (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT,
  period VARCHAR(10), -- 2024-01
  amount DECIMAL(10,2),
  status VARCHAR(20), -- pending/paid/overdue
  created_at TIMESTAMP
);

-- 每个租户数据库
-- 用户表 (租户内)
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL, -- 冗余便于快速查询
  username VARCHAR(50) NOT NULL,
  email VARCHAR(100),
  role_id BIGINT,
  status TINYINT DEFAULT 1,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  INDEX idx_tenant (tenant_id)
);
```

#### 5.2 业务流程

```
1. 租户注册 → 租户管理服务创建租户
2. 自动创建租户数据库实例
3. 初始化租户数据库（表结构、默认数据）
4. 配置租户域名/子域名
5. 租户管理员激活账户
6. 租户管理员添加用户、分配角色
7. 终端用户登录（带 Tenant Context）
8. 请求通过网关注入 Tenant ID
9. 服务根据 Tenant ID 路由到对应数据库
10. 业务处理、写入租户数据库
11. 计费服务统计用量、生成账单
```

### 六、关键架构决策

#### ADR-001: 租户隔离模式

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 独立数据库** | 数据完全隔离、安全性最高 | 资源成本高、运维复杂 |
| **方案 B: 独立 Schema** | 成本较低、隔离较好 | 共享资源可能影响 |
| **方案 C: 共享数据库独立行** | 成本最低 | 隔离性最弱、有泄露风险 |

**最终选择**：方案 A - 独立数据库

**原因**：
- 1000+ 租户，大型规模，需要最高级别隔离
- 通用 SaaS，可能涉及敏感企业数据
- 成本可接受（云数据库价格下降）

**升级时机**：
- 租户规模超过 10000 需要考虑混合模式
- 非核心业务可降级到独立 Schema

---

#### ADR-002: 数据库架构

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 每个租户独立 MySQL 实例** | 完全隔离、性能好 | 资源浪费、管理复杂 |
| **方案 B: 独立数据库 + 连接池共享** | 资源利用率高 | 隔离性稍弱 |
| **方案 C: RDS 多实例** | 托管运维、自动备份 | 成本较高 |

**最终选择**：方案 C - 云厂商 RDS 多实例 + Tenant Proxy

**原因**：
- 1000+ 租户，独立实例运维成本高
- 云 RDS 托管降低成本
- Tenant Proxy 统一管理连接

---

#### ADR-003: 租户识别方式

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 子域名** | 用户感知好、SEO友好 | 需要 DNS 配置 |
| **方案 B: Header 传递** | 灵活、可在网关层添加 | 前端需要处理 |
| **方案 C: 路径区分** | 实现简单 | URL 不够美观 |

**最终选择**：方案 A + 方案 B 混合

- 用户通过子域名访问
- 网关提取 Tenant ID 注入 Header
- API 使用 Header 进行路由

---

#### ADR-004: 租户上下文传递

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: ThreadLocal** | 性能好、代码简洁 | 异步场景需特殊处理 |
| **方案 B: Context 传递** | 显式传递、异步安全 | 代码侵入性大 |
| **方案 C: 中间件注入** | 统一处理 | 学习成本 |

**最终选择**：方案 A + Go Context 混合

- HTTP 请求：Middleware 提取 Tenant ID 到 Context
- 服务内部：通过 Context 传递
- 异步任务：显式传递 Tenant ID

---

#### ADR-005: 计费模式

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 预付费套餐** | 收入可预测 | 用户可能有压力 |
| **方案 B: 按量付费** | 灵活、用户接受 | 计费复杂 |
| **方案 C: 混合模式** | 兼顾双方 | 系统复杂 |

**最终选择**：方案 C - 混合模式

- 基础功能：预付费套餐
- 超出配额：按量计费
- 支持升级/降级

---

#### ADR-006: 部署架构

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 每个租户独立集群** | 完全隔离 | 资源浪费 |
| **方案 B: 共享集群 + Namespace 隔离** | 资源利用率高 | 隔离性依赖 K8s |
| **方案 C: Serverless** | 免运维、按需 | 冷启动延迟 |

**最终选择**：方案 B - 共享集群 + Namespace + 资源配额

**原因**：
- 1000+ 租户，独立集群成本太高
- K8s Namespace 隔离 + 资源配额
- 关键租户可使用独立节点池

### 七、风险与演进

#### 当前风险

| 风险类型 | 具体风险 | 应对措施 |
|----------|----------|----------|
| **性能风险** | 1000+ 数据库连接数爆炸 | 数据库代理、连接池共享 |
| **安全风险** | 租户间数据泄露 | 严格 SQL 审计、租户 ID 强制检查 |
| **运维风险** | 租户数据库维护困难 | 自动化备份、Tenant Operator |
| **成本风险** | 数据库资源浪费 | 按需分配、用量监控 |

#### 演进路线

| 阶段 | 规模 | 架构演进 |
|------|------|----------|
| **阶段 1** | 0-100 租户 | 独立数据库、手动管理 |
| **阶段 2** | 100-1000 租户 | Tenant Operator 自动化 |
| **阶段 3** | 1000-5000 租户 | 数据库分级（核心/普通） |
| **阶段 4** | 5000+ 租户 | 多区域部署、Serverless 混合 |

#### 扩展点预留

- 支持租户自定义字段
- 支持租户间数据导出（管理员授权）
- 支持租户数据回收（到期保留期）
- 支持多语言/多时区

### 九、落地建议

#### 实施路线图

| 阶段 | 时间 | 任务 |
|------|------|------|
| **第 1-2 周** | 2 周 | 平台基础设施：K8s、Kong、Tenant Proxy |
| **第 3-4 周** | 2 周 | 租户管理服务：租户注册、数据库创建 |
| **第 5-6 周** | 2 周 | 认证权限服务：多租户 Context 处理 |
| **第 7-8 周** | 2 周 | 计费服务：套餐管理、用量统计 |
| **第 9-10 周** | 2 周 | 业务服务：用户/订单多租户改造 |
| **第 11-12 周** | 2 周 | Tenant Operator、自动化运维 |
| **第 13-14 周** | 2 周 | 测试、压测、监控告警完善 |

#### 实施优先级

| 优先级 | 模块 | 原因 |
|--------|------|------|
| **P0** | 租户管理 + Tenant Proxy | 核心隔离能力 |
| **P0** | 认证权限服务 | 安全基础 |
| **P1** | 计费服务 | 商业变现 |
| **P2** | 业务服务多租户改造 | 业务能力 |
| **P3** | Tenant Operator | 运维自动化 |

#### 关键注意事项

1. **租户 ID 必须贯穿全链路** - 从网关到数据库
2. **数据库连接要严格隔离** - Tenant Proxy 是关键
3. **计费要准确** - 提前设计计量埋点
4. **自动化是关键** - 1000+ 租户无法手动管理

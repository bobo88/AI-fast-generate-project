# 企业监控系统架构研究

## 主题：企业级全栈监控系统架构设计

---

### 一、背景与目标

#### 背景

随着公司业务发展，线上服务数量逐渐增多，当前缺乏统一的监控平台，存在以下问题：

- 各项目自行搭建监控，标准不统一，维护成本高
- 问题发现滞后，往往是用户反馈后才知晓
- 缺少全局视角，无法快速定位问题根因
- 告警分散，运维人员无法第一时间收到通知

#### 当前问题

1. **监控碎片化** - 每个项目单独搭建监控，重复投入
2. **响应慢** - 故障发现依赖人工或用户反馈
3. **定位难** - 缺少链路追踪和问题关联分析
4. **通知分散** - 没有统一的告警通道

#### 架构目标

| 目标类型 | 具体目标 |
|----------|----------|
| **功能目标** | 支持全栈监控（HTTP/主机/容器/DB）、统一告警、仪表盘展示 |
| **性能目标** | 秒级数据采集、毫秒级查询响应 |
| **安全目标** | 采集代理安全认证、数据传输加密 |
| **可维护目标** | 配置化接入、新增监控项无需改代码 |
| **可扩展目标** | 支持从 50 个服务扩展到 500+ 服务 |

### 二、系统边界

#### 系统边界

| 类别 | 内容 |
|------|------|
| **输入** | 被监控服务的指标数据（HTTP 状态码/响应时间/服务器资源/容器状态/数据库性能） |
| **输出** | 监控仪表盘、告警通知、趋势图表 |
| **外部系统** | 钉钉/企业微信/飞书（告警通知）、被监控的各业务服务 |
| **内部模块** | 数据采集、数据存储、告警引擎、Web 控制台 |
| **不包含范围** | 日志收集与分析、链路追踪（可后期扩展） |

#### 上下游关系

```

└─────────────────┘     └──────────────────┘     └─────────────────┘
                               │
                               ▼
                        ┌──────────────────┐

                        └──────────────────┘
```

### 三、架构分层

#### 整体架构分层

```md
#### 用户层

- Web 管理后台（Vue3）
- 仪表盘展示页
- 告警配置页
- 设置中心

#### 接入层

- Nginx（反向代理）
- API Gateway（统一入口）

#### 业务层

- 采集服务（Agent/Exporter）
- 告警引擎
- 数据处理服务
- 配置中心

#### 数据层

- PostgreSQL（元数据/配置）
- Prometheus（时序指标数据）
- Redis（缓存/限流）

#### 基础设施层

- Docker（单机部署）
- Prometheus Server
- Grafana（可选，作为仪表盘）
```

#### 每层职责与交互

| 层级 | 职责 | 上下游交互 |
|------|------|------------|
| 用户层 | 展示监控数据、配置告警规则 | 通过 API 与业务层交互 |
| 接入层 | 请求路由、负载均衡、SSL 终结 | 分发请求到业务层服务 |
| 业务层 | 数据采集、存储、告警判断、配置管理 | 读写数据层、发送告警 |
| 数据层 | 持久化指标数据、元数据、缓存 | 被业务层访问 |
| 基础设施层 | 提供运行时环境、监控能力 | 部署业务层和数据层组件 |

### 四、核心模块设计

#### 4.1 采集模块 (Collector)

```
职责：负责从被监控目标收集指标数据
为什么存在：将监控逻辑与业务逻辑分离，统一采集标准
输入：被监控目标地址、采集间隔、指标类型
输出：标准化指标数据
依赖：被监控服务、时序数据库
核心数据：指标名称、标签、时间戳、数值
风险：采集延迟、目标不可达
扩展点：支持自定义 Exporter
```

#### 4.2 存储模块 (Storage)

```
职责：存储时序指标数据，提供查询能力
为什么存在：时序数据量大，需要高效存储和查询
输入：指标数据
输出：查询结果、聚合数据
依赖：PostgreSQL、Prometheus
核心数据：指标、标签、时间序列
风险：数据膨胀、查询性能下降
扩展点：支持数据压缩、长期归档
```

#### 4.3 告警引擎 (Alert Engine)

```
职责：判断指标是否触发告警规则，发送告警
为什么存在：实现自动化监控告警，减少人工值守
输入：告警规则、实时指标数据
输出：告警事件、通知消息
依赖：告警规则配置、通知渠道
核心数据：告警规则、告警历史、通知渠道配置
风险：告警风暴、规则漏配
扩展点：支持告警升级、抑制、聚合
```

#### 4.4 控制台 (Console)

```
职责：提供 Web 界面，展示监控数据、配置管理
为什么存在：人机交互入口，降低使用门槛
输入：用户操作请求
输出：页面数据、配置结果
依赖：后端 API
核心数据：仪表盘配置、告警规则、用户设置
风险：权限泄露
扩展点：支持自定义仪表盘、第三方集成
```

   │   Alert     │  │  Dashboard  │  │ Notification│
   │  (告警记录)  │  │  (仪表盘)   │  │ (通知记录)  │
   └─────────────┘  └─────────────┘  └─────────────┘

#### 核心表设计

```sql
-- 项目表
CREATE TABLE projects (
  id BIGINT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  description VARCHAR(255),
  created_at TIMESTAMP
);

-- 监控目标表
CREATE TABLE targets (
  id BIGINT PRIMARY KEY,
  project_id BIGINT REFERENCES projects(id),
  name VARCHAR(100),
  type VARCHAR(20), -- http/host/container/db
  endpoint VARCHAR(255),
  status TINYINT,
  created_at TIMESTAMP
);

-- 告警规则表
CREATE TABLE alert_rules (
  id BIGINT PRIMARY KEY,
  name VARCHAR(100),
  target_id BIGINT REFERENCES targets(id),
  metric_name VARCHAR(50),
  condition VARCHAR(50), -- > / < / =
  threshold FLOAT,
  duration INT, -- 持续时间(秒)
  severity VARCHAR(20), -- critical/warning/info
  enabled TINYINT
);

-- 告警记录表
CREATE TABLE alert_records (
  id BIGINT PRIMARY KEY,
  rule_id BIGINT REFERENCES alert_rules(id),
  target_id BIGINT,
  message TEXT,
  status VARCHAR(20), -- triggered/resolved
  fired_at TIMESTAMP,
  resolved_at TIMESTAMP
);

-- 通知配置表
CREATE TABLE notification_channels (
  id BIGINT PRIMARY KEY,
  type VARCHAR(20), -- dingtalk/wechat/email
  config JSONB,
  enabled TINYINT
);
```

#### 5.2 业务流程

```
1. 用户配置监控目标（HTTP接口/服务器/容器/数据库）
2. 配置采集间隔和指标类型
3. 采集服务按配置拉取指标数据
4. 数据写入时序数据库
5. 告警引擎周期性检查规则
6. 触发告警 → 发送通知 → 记录告警事件
7. 用户通过控制台查看仪表盘和告警历史
```

### 六、关键架构决策

#### ADR-001: 时序数据存储选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: MySQL/PostgreSQL** | 简单、团队熟悉 | 大数据量查询慢、存储成本高 |
| **方案 B: Prometheus** | 专为时序设计、查询高效、生态完善 | 单机版扩展性有限 |
| **方案 C: InfluxDB** | 时序领域专业、压缩率高 | 集群版收费 |

**最终选择**：方案 B - Prometheus 单机版

**原因**：
- 公司规模 <50 服务，数据量可控
- Prometheus 查询语言 PromQL 强大
- 与 Go 技术栈天然契合
- 社区成熟，集成生态丰富

**缺点**：
- 单机有数据量上限（约 100 万指标/秒）
- 未来扩展需考虑联邦或远程存储

**升级时机**：
- 当单机写入超过 50 万指标/秒
- 需要跨集群查询时

---

#### ADR-002: 告警通知渠道

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 邮件** | 无成本 | 及时性差 |
| **方案 B: 钉钉/企业微信** | 即时送达、免费 | 需要机器人配置 |
| **方案 C: 短信** | 及时 | 有成本 |

**最终选择**：方案 B - 钉钉/企业微信机器人

**原因**：
- 公司已在使用即时通讯工具
- 机器人配置简单、免费
- 支持卡片消息，展示效果好

---

#### ADR-003: 部署架构

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 单体部署** | 简单、运维成本低 | 扩展性有限 |
| **方案 B: Docker Compose** | 容器化、便于迁移 | 需要多容器管理 |
| **方案 C: Kubernetes** | 弹性伸缩、自动恢复 | 复杂度高、小规模不划算 |

**最终选择**：方案 A - 单体部署（Docker）

**原因**：
- 监控规模 <50 服务，单机足以承载
- 团队 K8s 经验有限，优先考虑简单方案
- 未来可平滑升级到 K8s

---

#### ADR-004: 前端技术选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: Vue3 + Element Plus** | 团队熟悉、组件丰富 | - |
| **方案 B: React + Ant Design** | 生态更大 | 团队无经验 |
| **方案 C: Grafana UI** | 专业监控界面 | 定制化难 |

**最终选择**：方案 A - Vue3 + Element Plus

**原因**：
- 团队已有 Vue3 经验
- Element Plus 组件丰富，适合后台管理系统
- 可复用现有项目模板

---

#### ADR-005: 采集方式

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 拉取 (Pull)** | 监控端无感、可控性好 | 需要开放端口 |
| **方案 B: 推送 (Push)** | 无需开放端口 | 需要改业务代码 |

**最终选择**：方案 A - Pull 模式为主

**原因**：
- Prometheus生态推荐Pull模式
- 可复用 exporters（node_exporter、mysql_exporter）
- 业务服务无需改造

### 七、风险与演进

#### 当前风险

| 风险类型 | 具体风险 | 应对措施 |
|----------|----------|----------|
| **性能风险** | 单机Prometheus数据量过大 | 评估数据量，提前规划联邦架构 |
| **安全风险** | 采集端点暴露 | 添加认证Token、启用HTTPS |
| **人员风险** | 核心开发离职 | 文档沉淀、代码规范 |
| **业务风险** | 告警风暴 | 配置告警抑制、聚合 |

#### 演进路线

| 阶段 | 规模 | 架构演进 |
|------|------|----------|
| **阶段 1** | 0-50 服务 | 单体部署 + Prometheus + 自研Web |
| **阶段 2** | 50-200 服务 | Prometheus联邦 + 引入Grafana |
| **阶段 3** | 200-500 服务 | 微服务化 + Thanos长期存储 |
| **阶段 4** | 500+ 服务 | 多集群 + 智能告警 |

#### 扩展点预留

- 预留 Exporter 扩展接口，支持自定义指标
- 告警规则支持模板化批量配置
- 数据可导出供后续大数据分析

- 先用已有开源方案（Prometheus + Grafana）验证需求

- 采集端可先使用已有的 exporters

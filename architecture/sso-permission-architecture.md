# SSO 统一认证与权限架构

## 主题：企业级 SSO 统一认证与 RBAC+ABAC 权限架构设计

---

### 一、背景与目标

#### 背景

开放平台需要支持：
- 第三方应用通过 OAuth2 接入
- 多租户场景下的统一身份认证
- 细粒度的资源权限控制（RBAC + ABAC）
- 支持 JWT 无状态认证

#### 当前问题

1. **认证分散** - 各业务系统独立开发认证，标准不统一
2. **权限混乱** - 缺乏统一的权限模型，权限控制分散
3. **第三方接入难** - 缺乏标准 OAuth2 接入流程
4. **安全风险** - 认证实现不规范，存在安全漏洞

#### 架构目标

| 目标类型 | 具体目标 |
|----------|----------|
| **功能目标** | 统一认证中心、OAuth2 授权、RBAC+ABAC 权限 |
| **性能目标** | 认证请求 <50ms、支持万级并发 |
| **安全目标** | Token 加密、刷新机制、审计日志 |
| **可维护目标** | 认证配置化、权限可视化 |
| **可扩展目标** | 支持多种认证协议、第三方应用接入 |

### 二、系统边界

#### 系统边界

| 类别 | 内容 |
|------|------|
| **输入** | 用户登录请求、第三方应用授权请求、API 访问请求 |
| **输出** | JWT Token、授权码、权限校验结果 |
| **外部系统** | 第三方应用（Client）、LDAP/AD、短信/邮件服务 |
| **内部模块** | 认证服务、授权服务、权限服务、用户服务、Token 服务 |
| **不包含范围** | 业务系统内部权限（仅提供接口） |

### 三、架构分层

#### 整体架构分层

```
#### 用户层

- 登录页面 (Web)
- 授权确认页 (OAuth2)
- 用户中心 (个人信息、修改密码)
- 管理员后台 (用户管理、权限配置)

#### 接入层

- Nginx (负载均衡)
- API Gateway (统一入口、Token 校验)
- WAF (安全防护)

#### 业务层

├── 认证服务 (Auth Service)
│   ├── 用户名密码登录
│   ├── 手机验证码登录
│   ├── 第三方登录 (OAuth2)
│   └── JWT Token 签发与校验
│
├── 授权服务 (Authorization Service)
│   ├── OAuth2 授权服务器
│   ├── 授权码/Token 管理
│   └── 第三方应用管理
│
├── 权限服务 (Permission Service)
│   ├── RBAC 权限管理
│   ├── ABAC 策略引擎
│   └── 资源权限校验
│
└── 用户服务 (User Service)
    ├── 用户 CRUD
    ├── 组织架构管理
    └── 用户注册/激活

#### 数据层

├── MySQL (用户、角色、权限数据)
├── Redis (Token 缓存、验证码)
└── LDAP (可选，企业 AD 集成)

#### 基础设施层

├── Docker / Kubernetes
├── CI/CD
└── 日志/监控
```

#### 每层职责与交互

| 层级 | 职责 | 关键能力 | 禁止跨层 |
|------|------|----------|----------|
| 用户层 | 登录/授权 UI | 多端适配 | - |
| 接入层 | 请求路由、Token 校验 | 网关鉴权 | 禁止绕过网关 |
| 业务层 | 认证/授权/权限逻辑 | 服务自治 | 业务服务禁止直接查用户表 |
| 数据层 | 数据持久化 | 数据隔离 | - |
| 基础设施层 | 运行环境 | 弹性伸缩 | - |

### 四、核心模块设计

#### 4.1 认证服务 (Auth Service)

```
职责：用户身份认证、Token 签发
为什么存在：统一认证入口、规范认证流程
输入：用户凭证（用户名密码/验证码/第三方Token）
输出：JWT Access Token + Refresh Token
依赖：用户服务、Redis、加密服务
核心数据：用户ID、Token 内容、过期时间
风险：Token 泄露、暴力破解
扩展点：支持多种登录方式、MFA 增强
```

#### 4.2 授权服务 (OAuth2 Provider)

```
职责：OAuth2 授权服务器、第三方应用管理
为什么存在：开放平台标准、支持第三方应用接入
输入：第三方应用授权请求
输出：授权码、Access Token、Refresh Token
依赖：Auth Service、Client 注册数据
核心数据：Client ID/Secret、授权范围、Token 策略
风险：授权码泄露、Scope 过度授权
扩展点：支持 OAuth2.1、PKCE 流程
```

#### 4.3 权限服务 - RBAC 模块

```
职责：基于角色的访问控制
为什么存在：粗粒度权限管理、角色批量授权
输入：用户ID、角色ID、资源请求
输出：是否拥有权限
依赖：用户服务、角色数据、权限数据
核心数据：用户-角色映射、角色-权限映射
风险：角色权限过度、权限蔓延
扩展点：支持角色继承、角色组
```

#### 4.4 权限服务 - ABAC 模块

```
职责：基于属性的动态权限控制
为什么存在：细粒度动态权限、上下文感知
输入：用户属性、资源属性、环境属性
输出：是否允许访问
依赖：RBAC 数据、属性配置、策略引擎
核心数据：访问策略、属性规则、上下文
风险：策略配置复杂、规则冲突
扩展点：支持复杂策略表达式
```

#### 4.5 Token 服务

```
职责：Token 管理、刷新、撤销
为什么存在：JWT 无状态、需支持续期和撤销
输入：Token 刷新请求、撤销请求
输出：新的 Token、撤销结果
依赖：Redis（Token 黑名单）、加密服务
核心数据：Token 签发记录、黑名单
风险：Token 泄露无法撤销
扩展点：支持 Token 类型的细粒度控制
```

#### 4.6 用户服务

```
职责：用户生命周期管理、组织架构
为什么存在：统一用户数据源
输入：用户注册/更新请求
输出：用户信息、组织架构
依赖：LDAP（可选）、主数据系统
核心数据：用户信息、部门/组织、用户组
风险：用户数据不一致
扩展点：支持多租户、用户同步
```

#### 核心表设计

```sql
-- 用户表
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  password_hash VARCHAR(255),
  email VARCHAR(100),
  phone VARCHAR(20),
  status TINYINT DEFAULT 1, -- 1=active, 0=disabled
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- 角色表
CREATE TABLE roles (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50) NOT NULL,
  code VARCHAR(50) UNIQUE NOT NULL,
  description VARCHAR(255),
  status TINYINT DEFAULT 1,
  created_at TIMESTAMP
);

-- 权限表
CREATE TABLE permissions (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50) NOT NULL,
  code VARCHAR(100) UNIQUE NOT NULL, -- permission:user:read
  type VARCHAR(20), -- menu/button/api
  resource VARCHAR(100), -- 资源标识
  action VARCHAR(20), -- read/write/delete
  description VARCHAR(255)
);

-- 用户角色关联
CREATE TABLE user_roles (
  user_id BIGINT,
  role_id BIGINT,
  PRIMARY KEY (user_id, role_id)
);

-- 角色权限关联
CREATE TABLE role_permissions (
  role_id BIGINT,
  permission_id BIGINT,
  PRIMARY KEY (role_id, permission_id)
);

-- ABAC 策略表
CREATE TABLE policies (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50),
  effect VARCHAR(10), -- allow/deny
  subject JSONB, -- 用户属性条件
  resource JSONB, -- 资源条件
  action JSONB, -- 操作条件
  condition JSONB, -- 额外条件
  priority INT DEFAULT 0
);

-- OAuth2 客户端
CREATE TABLE oauth_clients (
  id BIGINT PRIMARY KEY,
  client_id VARCHAR(50) UNIQUE,
  client_secret VARCHAR(100),
  name VARCHAR(100),
  redirect_uris JSONB,
  scopes JSONB,
  grant_types JSONB,
  created_at TIMESTAMP
);

-- Token 黑名单
CREATE TABLE token_blacklist (
  token_hash VARCHAR(64) PRIMARY KEY,
  expire_at TIMESTAMP
);
```

#### 5.2 认证流程 (JWT)

```
1. 用户提交登录凭证（用户名密码）
2. 认证服务校验凭证
3. 校验通过 → 生成 JWT (Access Token + Refresh Token)
4. 返回 Token 给客户端
5. 客户端携带 Token 访问业务 API
6. API Gateway 校验 Token 签名和有效期
7. 校验通过 → 路由到业务服务
8. 业务服务通过权限服务校验权限
9. 返回业务数据
```

#### 5.3 OAuth2 授权流程

```
1. 第三方应用引导用户跳转授权页面
2. 用户确认授权
3. 授权服务生成授权码 (code)
4. 第三方应用用 code 换取 Token
5. 授权服务验证后发放 Access Token
6. 第三方应用携带 Token 调用资源 API
7. API 校验 Token 并返回资源
```

```

│ 第三方应用│────▶│ 授权服务    │────▶│   Token服务  │────▶│  Redis   │
│          │     │ (OAuth2)    │     │  (签发Token) │     │ (缓存)   │
└──────────┘     └─────────────┘     └──────────────┘     └───────────┘

                 │  (JWT签发)   │                              │  (用户数据)│
```

### 六、关键架构决策

#### ADR-001: Token 方案选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: JWT 无状态** | 性能高、无需存储、分布式友好 | 撤销困难、Payload 有限 |
| **方案 B: Session 有状态** | 可控性强、易于撤销 | 存储开销、分布式需共享 |
| **方案 C: Redis 缓存 Token** | 可控性强、性能好 | 需要额外存储 |

**最终选择**：方案 A + 方案 C 混合

- Access Token：JWT 无状态（高频访问）
- Refresh Token：存储 Redis（支持撤销）
- Token 黑名单：Redis 存储（及时失效）

**原因**：JWT 性能好，配合黑名单实现可控撤销

---

#### ADR-002: 权限模型选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 纯 RBAC** | 简单、易理解 | 粒度粗、无法动态 |
| **方案 B: 纯 ABAC** | 细粒度、动态 | 复杂、难以维护 |
| **方案 C: RBAC + ABAC** | 兼顾简单与灵活 | 需要协调两者 |

**最终选择**：方案 C - RBAC 做角色基础 + ABAC 做动态增强

- 角色分配：RBAC（粗粒度）
- 动态权限：ABAC（时间、IP、环境）
- 资源级别：ABAC（数据权限）

---

#### ADR-003: OAuth2 实现方式

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 自研 OAuth2** | 完全可控 | 开发工作量大 |
| **方案 B: Keycloak** | 功能全面、企业级 | 资源重、定制难 |
| **方案 C: Casbin + 自研** | 轻量、可定制 | 需要开发 |

**最终选择**：方案 C - 基于 Go + Casbin 自研

**原因**：
- 团队 Go 技术栈
- Casbin 成熟、轻量
- 可控性强、定制灵活

---

#### ADR-004: 密码存储

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 明文** | - | 极度不安全 |
| **方案 B: MD5/SHA** | 简单 | 易被彩虹表破解 |
| **方案 C: bcrypt** | 安全自适应 | 性能稍慢 |
| **方案 D: Argon2** | 最安全、内存友好 | 需 Go 1.20+ |

**最终选择**：方案 D - Argon2

**原因**：
- Go 1.21+ 支持
- 当前最安全的密码哈希算法
- 抗GPU破解

---

#### ADR-005: API 网关鉴权

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 网关集中鉴权** | 统一入口、逻辑简单 | 网关压力大 |
| **方案 B: 各服务独立鉴权** | 分散压力 | 重复代码、难以统一 |
| **方案 C: 混合模式** | 平衡两者 | 复杂度高 |

**最终选择**：方案 A - 网关集中鉴权

**原因**：
- JWT 验签性能开销小
- 统一安全策略
- 便于日志审计

---

#### ADR-006: 密码登录 vs 免密登录

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案 A: 仅密码登录** | 传统、用户熟悉 | 密码安全风险 |
| **方案 B: 密码 + 验证码** | 双重验证 | 流程复杂 |
| **方案 C: 密码 + MFA** | 最高安全 | 用户体验差 |

**最终选择**：方案 B - 密码 + 短信/邮箱验证码

**原因**：
- 平衡安全与体验
- 敏感操作二次验证
- 支持找回密码

### 七、风险与演进

#### 当前风险

| 风险类型 | 具体风险 | 应对措施 |
|----------|----------|----------|
| **安全风险** | Token 泄露被滥用 | 短过期时间 + 黑名单 |
| **安全风险** | 暴力破解密码 | 限流 + 验证码 + MFA |
| **性能风险** | 权限校验频繁 | Redis 缓存 + 本地缓存 |
| **运维风险** | 权限配置混乱 | 权限可视化 + 审计日志 |

#### 演进路线

| 阶段 | 规模 | 架构演进 |
|------|------|----------|
| **阶段 1** | 初创 | 基础认证 + 简单 RBAC |
| **阶段 2** | 成长期 | JWT + OAuth2 + RBAC |
| **阶段 3** | 成熟期 | RBAC + ABAC + MFA |
| **阶段 4** | 大型 | 分布式认证 + 零信任 |

#### 扩展点预留

- 支持 MFA（短信、邮箱、硬件）
- 支持 Social Login（Google、GitHub）
- 支持 LDAP/AD 集成
- 支持 Session 并发控制

### 九、落地建议

#### 实施路线图

| 阶段 | 时间 | 任务 |
|------|------|------|
| **第 1 周** | 1 周 | 基础框架搭建、用户服务 |
| **第 2 周** | 1 周 | 认证服务（密码登录 + JWT） |
| **第 3 周** | 1 周 | 权限服务 RBAC 实现 |
| **第 4 周** | 1 周 | 权限服务 ABAC 策略引擎 |
| **第 5 周** | 1 周 | OAuth2 授权服务 |
| **第 6 周** | 1 周 | API 网关集成 |
| **第 7 周** | 1 周 | 安全增强（MFA、限流） |
| **第 8 周** | 1 周 | 文档、测试、上线 |

#### 实施优先级

| 优先级 | 模块 | 原因 |
|--------|------|------|
| **P0** | 认证服务 + JWT | 核心认证能力 |
| **P0** | 权限服务 RBAC | 基础权限控制 |
| **P1** | OAuth2 授权服务 | 第三方接入 |
| **P1** | 权限服务 ABAC | 增强权限 |
| **P2** | MFA / 安全增强 | 安全加固 |

#### 关键注意事项

1. **Token 安全** - 短期 Access Token + Refresh Token
2. **密码安全** - 使用 Argon2 哈希
3. **权限校验** - 每次 API 调用都必须校验
4. **审计日志** - 记录所有敏感操作

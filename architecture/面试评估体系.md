## 三、逐环节面试题库

### 环节1: 背景了解 (5分钟)

**目的**: 放松氛围，了解候选人基本信息

```
暖场问题:
1. 请简单介绍一下你自己，以及为什么对这个岗位感兴趣？
2. 你目前是在职还是离职状态？
3. 你的职业规划是什么？3年后希望达到什么状态？

快速判断:
- 表达是否清晰
- 对岗位是否有热情
- 职业规划是否清晰
```

---

### 环节2: 产品基础能力 (15分钟)

**核心问题**:

```markdown
1. 你认为产品经理最重要的能力是什么？为什么？
   追问: 能举一个具体例子说明吗？

2. 你做过的最复杂的产品是什么？请从背景、目标、用户、功能、结果完整讲一遍。
   追问:
   - 这个产品有多少人参与？
   - 你负责什么部分？
   - 做了多长时间？
   - 最后结果如何？

3. 当老板只告诉你"做一个远程控制机器人功能"，你会怎么开始？
   追问:
   - 你会问哪些问题？
   - 你会找哪些人沟通？
   - 第一步做什么？

4. 你通常如何整理需求？
   追问:
   - 需求来源有哪些？
   - 你怎么区分需求和功能？
   - 需求文档包含什么？

5. 一个需求从提出到上线，你具体会怎么推进？
   追问:
   - 每个阶段的产出是什么？
   - 你怎么保证进度？
   - 上线后你怎么验证效果？
```

**判断标准**:

| 级别 | 特征 |
|------|------|
| 初级 | 只会描述功能，不会讲目标、用户、场景 |
| 中级 | 能讲清楚完整流程，会拆需求，能说出优先级 |
| 高级 | 能从业务、用户、技术、成本一起考虑，能讲清楚为什么 |

---

### 环节3: 硬件/机器人理解 (12分钟)

**核心问题**:

```markdown
1. 你是否做过和硬件、IoT、机器人相关的产品？
   - 如果做过: 请详细介绍你负责的硬件产品
   - 如果没做过: 你对硬件产品有什么理解？

2. 如果让你设计一个可以远程控制的小车机器人，你会考虑哪些模块？
   追问:
   - 硬件部分需要哪些？
   - 软件部分需要哪些？
   - 云端需要什么？

3. 摄像头、云台、电机、电池、网络、GPS、避障分别会影响哪些产品能力？
   追问:
   - 如果要降低成本的优先级？

4. 如果用户反馈"机器人有时候控制会延迟 2 秒"，你会怎么排查？
   追问:
   - 你会找哪些人协助？
   - 可能的原因有哪些？
   - 产品层面能做什么？

5. 机器人在家里断网了，产品上应该怎么设计？
   追问:
   - 离线状态下用户能看到什么？
   - 机器人应该保持什么状态？
   - 恢复网络后怎么做？

6. 如果机器人电量只剩 10%，系统应该如何处理？
   追问:
   - 用户应该收到什么通知？
   - 机器人应该做什么动作？
   - 产品设计上要考虑哪些场景？
```

**候选人应提到关键词**:

- 网络延迟 / 弱网场景 / 离线机制
- 电量不足 / 自动回充
- 设备状态同步
- 摄像头与视频流
- 硬件故障 / 固件升级 / OTA
- 安全性

**高级追问**:

```
假设现在要做一个家庭巡逻机器人：
- 可以远程查看监控
- 可以移动
- 可以自动巡逻
- 可以异常报警

请你拆出：
1. 用户场景
2. 核心功能
3. MVP
4. 后续版本路线
5. 最大风险
```

---

### 环节4: 需求拆解与优先级 (10分钟)

**核心问题**:

```markdown
1. 现在有 20 个需求，但开发资源只够做 5 个，你怎么决定？
   追问:
   - 你依据什么排序？
   - 谁来决定优先级？
   - 怎么让其他人接受？

2. 如果老板、客户、研发都在提不同需求，你怎么平衡？
   追问:
   - 有没有冲突的需求？
   - 你怎么协调？
   - 最后谁让步？

3. 一个功能用户想要，但技术成本很高，你怎么处理？
   追问:
   - 你会怎么评估技术成本？
   - 有替代方案吗？
   - 你怎么说服用户或研发？

4. 你通常如何定义 MVP？
   追问:
   - MVP 怎么验证假设？
   - 最小功能集怎么确定？
   - 验证后怎么迭代？
```

**希望候选人提到**:

- 用户价值 / 商业价值
- 技术成本 / 实现难度 / 时间成本
- 风险 / ROI
- Kano模型 / 四象限 / RICE

**验证是否真实**:

```
请举一个你真的砍掉过的需求。
- 为什么砍？
- 最后谁不满意？
- 你怎么说服他们？
```

---

### 环节5: 结构化思维 (10分钟)

**现场命题**:

```
命题: "我们准备做一个机器人远程看家产品，请你现场拆解。"
时间: 10 分钟
要求: 候选人现场在白板/纸上写出结构
```

**观察要点**:

候选人是否能主动拆成:
- [ ] 用户是谁
- [ ] 场景是什么
- [ ] 核心问题是什么
- [ ] 主要功能是什么
- [ ] MVP 是什么
- [ ] 风险是什么

**高级标准**:

优秀候选人会主动输出:
```
用户 → 场景 → 问题 → 需求 → 功能 → 流程 → 风险 → 数据指标
```

**注意**: 如果候选人一上来就直接说功能，多半不够成熟

---

### 环节6: 数据与结果导向 (8分钟)

**核心问题**:

```markdown
1. 你做过的产品里，最重要的指标是什么？
   追问:
   - 为什么是这个指标？
   - 这个指标怎么计算？
   - 目标值是多少？

2. 你如何判断一个功能是否成功？
   追问:
   - 你看哪些数据？
   - 数据从哪里来？
   - 成功的标准是什么？

3. 如果一个新功能上线后，用户并没有使用，你会怎么办？
   追问:
   - 你会分析什么？
   - 可能的原因有哪些？
   - 你会做什么决策？

4. 你会看哪些数据？
   追问:
   - 日常监控哪些指标？
   - 异常指标怎么发现？
   - 数据怎么指导产品决策？
```

**机器人相关指标**:

- 设备在线率
- 日活 (DAU)
- 使用时长
- 控制成功率
- 视频观看时长
- 报警触发率
- 留存率
- 复购率
- 故障率

**追问**:

```
你之前项目中，有没有哪个指标没有达到目标？
- 为什么？
- 你做了什么？
- 最后改善了吗？
```

---

### 环节7: 跨团队协作 (8分钟)

**核心问题**:

```markdown
1. 你合作过最难沟通的研发是什么情况？
   追问:
   - 具体是什么问题？
   - 你怎么解决的？
   - 最终结果如何？

2. 研发说做不了，你怎么办？
   追问:
   - 你会怎么沟通？
   - 有没有替代方案？
   - 怎么判断是真的做不了还是不想做？

3. 硬件延期、软件延期、测试延期时，你如何处理？
   追问:
   - 你会做什么？
   - 怎么协调各方？
   - 延期了怎么跟老板/客户交代？

4. 如果两个部门意见完全冲突，你怎么推动？
   追问:
   - 具体冲突是什么？
   - 你站在哪边？
   - 最后怎么解决的？
```

**重点观察**:

- 是否会推动 (不是只催)
- 是否会对齐目标
- 是否会拿数据和事实说话

---

### 环节8: 项目管理 (8分钟)

**核心问题**:

```markdown
1. 你通常如何排期？
   追问:
   - 怎么评估工作量？
   - 怎么分配资源？
   - 排期工具用什么？

2. 如何保证项目按时上线？
   追问:
   - 里程碑怎么设置？
   - 风险怎么识别？
   - 进度怎么跟踪？

3. 如果已经快上线了，但发现有一个严重问题，你会怎么办？
   追问:
   - 问题的影响是什么？
   - 你会怎么决策？
   - 怎么处理与上线时间的冲突？

4. 你是否做过跨月、跨季度的大项目？
   追问:
   - 有多少人在项目里？
   - 你怎么协调？
   - 遇到过最大的困难是什么？
```

**追问**:

```
- 具体有哪些里程碑？
- 谁负责什么？
- 哪个节点最容易延期？
- 如何提前发现风险？
```

---

### 环节9: 真实水平验证 (12分钟)

#### 方法一: 深挖经历

```
你说你做过 XX 功能。
那请你:
- 当时用户是谁？
- 最开始的问题是什么？
- 你亲自做了什么？
- 研发多少人？
- 做了多久？
- 最后结果是什么？

如果候选人答不出来，大概率不是核心参与者
```

#### 方法二: 现场设计

```
请你在 10 分钟内，现场设计一个"机器人远程看护系统"。

要求输出:
- 用户
- 场景
- 功能
- 页面
- 流程
- 风险
- 指标

看其结构化能力
```

#### 方法三: 反向验证

```
你刚才说这个功能很成功。
- 那它失败的可能性是什么？
- 如果重做一次，你会改什么？

真正做过的人，一定知道缺点和问题
```

---

### 环节10: 反向提问 (5分钟)

```
常规问题:
1. 你有什么想了解我们公司的？
2. 你对这个岗位有什么期望？
3. 你的期望薪资是多少？
4. 你最快什么时候可以入职？
```

## 四、面试评分表

### 4.1 评分维度

| 序号 | 能力维度 | 权重 | 1分 | 2分 | 3分 | 4分 | 5分 |
|:----:|----------|:----:|-----|-----|-----|-----|-----|
| 1 | 产品基础能力 | 15% | 只会描述功能 | 能讲流程 | 能拆需求 | 能说清为什么 | 能从多维度考虑 |
| 2 | 结构化思维 | 15% | 无逻辑 | 有一点逻辑 | 能拆解常见问题 | 拆解清晰有体系 | 主动输出完整结构 |
| 3 | 硬件/机器人理解 | 15% | 完全不懂 | 了解一点 | 有相关经验 | 理解深入 | 有完整硬件产品经验 |
| 4 | 需求拆解 | 10% | 不会拆 | 简单拆分 | 能分优先级 | 有方法论 | 能平衡多方需求 |
| 5 | 优先级能力 | 10% | 不会判断 | 凭感觉判断 | 有一定依据 | 有方法论 | 能做ROI评估 |
| 6 | 数据意识 | 10% | 不看数据 | 看一下数据 | 会分析数据 | 有数据驱动思维 | 能用数据指导决策 |
| 7 | 沟通推进 | 10% | 不会沟通 | 能沟通 | 能推动项目 | 能解决冲突 | 能协调多方资源 |
| 8 | 项目管理 | 5% | 不会管理 | 能跟进 | 能排期 | 能控制风险 | 能管理大项目 |
| 9 | 学习能力 | 5% | 不主动学习 | 被动学习 | 主动学习 | 快速学习 | 持续学习成长 |
| 10 | 与公司匹配度 | 5% | 不匹配 | 一般匹配 | 比较匹配 | 非常匹配 | 完美匹配 |

### 4.2 加分项

| 序号 | 加分项 | 分值 | 说明 |
|:----:|--------|:----:|------|
| 1 | 有机器人/硬件产品经验 | +2 | 直接相关经验 |
| 2 | 有从0到1项目经验 | +1 | 创业或新产品 |
| 3 | 有从1到100经验 | +1 | 产品增长阶段 |
| 4 | 有带团队经验 | +1 | 管理经验 |
| 5 | 有跨部门协调经验 | +1 | 多方协作能力 |

### 4.3 减分项

| 序号 | 减分项 | 分值 | 说明 |
|:----:|--------|:----:|------|
| 1 | 简历与面试差距大 | -2 | 夸大经历 |
| 2 | 对过往项目不熟悉 | -1 | 非核心参与 |
| 3 | 对行业不感兴趣 | -1 | 热情不足 |
| 4 | 逻辑混乱 | -1 | 表达不清 |
| 5 | 过于自信/自大 | -1 | 沟通问题 |

### 4.4 评分计算

```
综合得分 = Σ(能力维度得分 × 权重) + 加分项 - 减分项

最终等级:
- 4.5-5 分: 非常强，可以做核心 PM
- 3.5-4.5 分: 能独立负责模块
- 2.5-3.5 分: 需要培养
- 2.5 以下: 不建议录用
```

## 七、面试注意事项

### 7.1 面试原则

1. **禁止只问"懂不懂""做过没有"** - 必须通过经历→细节→决策→数据→复盘→追问来验证
2. **所有问题必须能继续向下追 3 层**
3. **每个能力点都要有真实案例验证**
4. **不看候选人讲得多好，看是否: 讲得清、有逻辑、有数字、能说出为什么**

### 7.2 注意事项

| 事项 | 说明 |
|------|------|
| 面试时间 | 建议 90 分钟左右，不要超过 2 小时 |
| 环境 | 安静、舒适、避免打断 |
| 记录 | 全程记录候选人回答要点 |
| 追问 | 遇到关键点一定要深挖 |
| 现场设计 | 必须让候选人现场动笔/动脑 |
| 反向验证 | 必须问失败案例和改进空间 |

### 7.3 常见候选人类型

| 类型 | 特征 | 应对策略 |
|------|------|----------|
| "背答案"型 | 回答流利但缺乏细节 | 深挖细节，问具体数字 |
| "夸大经历"型 | 简历丰富但答不出细节 | 深挖经历，问参与度 |
| "理论派"型 | 满口理论但没做过 | 问真实案例和结果 |
| "执行派"型 | 只说做了什么不说为什么 | 追问决策依据和结果 |
| "全能型"型 | 什么问题都能回答 | 反向验证，问失败案例 |
